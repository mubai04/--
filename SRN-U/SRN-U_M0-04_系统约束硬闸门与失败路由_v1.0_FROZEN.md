# 1. 文档目标

本文件回答五个问题：

1. 哪些条件不满足时，系统必须停止？
2. 哪些问题可以继续运行，但必须留下警告？
3. 问题出现后，应返回哪一层修复？
4. 什么情况下必须人工裁决？
5. 如何防止系统用低层润色掩盖高层结构错误？

本文件不负责：

- 具体代码实现；
- 具体检测算法；
- 具体模型选择；
- 具体评分公式；
- 具体数据库设计；
- 平台推荐预测；
- 读者心理精确测量。

# 2. 约束体系总览

SRN-U 的约束分为五类：

```text
1. 全局不变量
2. 硬闸门
3. 软检查
4. 控制信号
5. 人工审批门
```

它们的关系：

```text
全局不变量
    │
    ├─ 被违反 → BLOCKED
    │
硬闸门
    │
    ├─ 不通过 → 返回指定层修复
    │
软检查
    │
    ├─ 发现问题 → 警告、REWRITE或局部调整
    │
控制信号
    │
    ├─ 指挥运行路由，不等于对象状态
    │
人工审批门
    │
    └─ 重大变更必须人工批准
```

状态提交采用事务原则：

```text
预提交校验
→ 全部条件通过
→ 原子提交
→ 生成不可变快照
```

在预提交失败时，不得创建冻结快照，也不得替换当前活动状态。

# 3. 全局约束优先级

任何模块、指标、平台要求或审美偏好发生冲突时，必须按以下顺序裁决：

```text
P1 已发生事实与冻结真源
P2 世界硬规则
P3 人物核心动机、人格与知识边界
P4 因果关系、能力来源与资源条件
P5 不可逆后果和已承担代价
P6 当前主路径与章节目标
P7 信息隐藏、释放与悬念
P8 语言、节奏和文风
P9 符号、传播与平台适配
```

低优先级不得覆盖高优先级。

示例：

```text
不能为了章末钩子制造无来源危机
不能为了节奏让人物突然失去常识
不能为了悬念让角色故意不说正常信息
不能为了文风改掉已发生事实
不能为了平台适配改变作品核心承诺
```

# 4. 全局不变量

全局不变量是任何阶段都不得违反的系统底线。

## INV-01｜项目隔离

所有正式核心对象必须携带相同 `project_id`。

尚未对象化的原始输入必须满足以下二者之一：

```text
已经绑定当前项目上下文
或
明确声明为“创建新项目”
```

创建新项目时，由G0先分配或确认 `project_id`，随后A层创建正式 ProjectBrief。该入口上下文不是第十类核心对象。

禁止：

- 不同小说人物互相进入；
- 不同小说设定互相污染；
- 测试案例状态进入正式项目；
- 一个项目读取另一个项目快照。

违反结果：

```text
BLOCKED
→ H层隔离检查
→ B层重建有效状态
```

---

## INV-02｜唯一真源

同一项目、同一状态节点只能有一个当前有效真源。

必须区分：

```text
当前有效版本
历史冻结版本
被替代版本
候选版本
草稿版本
```

禁止：

- 两份文件同时声称是当前正文真源；
- 两个 NarrativeState 同时处于 active；
- 未标记版本关系的同名对象并存；
- 用聊天记录替代正式冻结文件。

违反结果：

```text
BLOCKED
→ H层真源核查
→ 人工裁决
```

---

## INV-03｜事实、计划、候选和草稿分离

以下内容不得混用：

```text
事实：NarrativeState / CharacterState
计划：ChapterBlueprint
候选：CandidatePath
正文草稿：ChapterDraft
冻结检查点：StateSnapshot
```

禁止：

- 把未来章纲写入 confirmed_facts；
- 把候选路径后果当成既定事实；
- 把未通过审计的草稿回写状态；
- 把模型推测写成人物已知信息。

违反结果：

```text
BLOCKED
→ B层状态清理
→ 必要时H层回滚
```

---

## INV-04｜来源可追溯

所有关键事实、规则、能力、资源、知识和重大关系变化必须有来源。

有效来源包括：

```text
ProjectSeed
已批准 ChapterDraft
上一 StateSnapshot
人工裁决
```

其中：

```text
PASS AuditReport
```

只证明指定 ChapterDraft 已通过审计，不能单独产生故事事实。章节事实进入状态时，必须同时引用：

```text
已批准 ChapterDraft
+
与该版本精确匹配的 PASS AuditReport
```

无来源内容只能标记为：

```text
UNKNOWN
UNDECIDED
DISPUTED
系统建议
```

不得标记为已确认事实。

---

## INV-05｜人物知识边界

角色只能根据自己已知、怀疑、误解或可现场推断的内容行动。

禁止：

- 使用作者知道但角色不知道的信息；
- 使用读者知道但角色不知道的信息；
- 读取其他角色私密信息；
- 无推理过程直接得出正确答案。

违反结果：

```text
硬失败
→ 若仅正文表达错误：REWRITE
→ 若蓝图已要求泄漏：REPLAN
→ 若状态本身冲突：BLOCKED
```

---

## INV-06｜能力与资源守恒

人物使用的能力、道具、权限、金钱、关系和时间必须有来源并受限制。

禁止：

- 凭空获得能力；
- 消耗品无限使用；
- 已丢失资源重新出现；
- 代价和伤势无原因消失；
- 时间与距离条件不成立。

违反结果：

```text
硬失败
→ E层或D层修复
→ 涉及状态错误时返回B层
```

---

## INV-07｜因果前置成立

进入 CandidatePath、ChapterBlueprint 或正式状态变更的关键事件必须满足必要前提。

关键事件至少要回答：

```text
谁做的
为什么做
凭什么能做
触发了什么
改变了什么
留下什么后果
```

若只能依靠“作者需要”解释，则判定为因果失效。

---

## INV-08｜不可逆变化继承

已经发生并被确认的不可逆变化不得静默恢复。

包括：

```text
死亡
公开暴露
关系破裂
资源永久损失
身份改变
承诺成立
法律或组织记录
已被他人得知的秘密
```

修正不可逆变化必须经过：

```text
新事实依据
+
审计确认
+
必要时人工审批
```

---

## INV-09｜审计独立

G层不能直接修改被审计对象。

必须遵循：

```text
发现问题
→ 生成 AuditReport
→ 路由到责任层修复
→ 产生新对象版本
→ G层重新复验
```

禁止：

```text
自己生成
→ 自己改
→ 自己宣布通过
```

---

## INV-10｜快照不可变

StateSnapshot 创建后，业务内容不得原地修改。

修正只能通过：

```text
生成新版本
标记旧版本 superseded / invalidated
保留完整引用链
```

---

## INV-11｜状态提交原子性

B-UPDATE 与 H-P2 之间必须保持以下事务语义：

```text
旧 NarrativeState 保持 active
新 NarrativeState 保持 initial
↓
G7与G8全部通过
↓
H-P2原子提交：
旧 active → superseded
新 initial → active
创建 StateSnapshot.frozen
selected CandidatePath → executed
```

在提交完成前：

- 不得同时存在两个 active NarrativeState；
- 不得提前冻结 StateSnapshot；
- 不得提前将旧状态标记为 superseded；
- 任一校验失败都必须保留旧活动状态不变。

---

## INV-12｜高层错误不得低层掩盖

失败必须返回拥有该问题修改权的层。

```text
项目定位错误 → A层
状态错误 → B层
路径错误 → C层
章纲错误 → D层
正文表达错误 → E层
生成中偏差 → F层提出建议，E/D层执行
审计错误 → G层
提交与快照错误 → H层
```

禁止：

- 用润色修复路径错误；
- 用新设定修复正文漏洞；
- 用删句掩盖人物动机不成立；
- 用章末危机掩盖本章没有状态变化。

---

## INV-13｜人工主权

SRN-U 可以建议，但不得自动决定以下事项：

```text
核心卖点改变
主角核心欲望改变
主要世界规则改变
主线删除或替换
已发布正文重写
重大人物死亡或复活
全局文风更换
项目商业方向改变
```

这些事项必须进入人工审批门。

# 5. 硬闸门总表

SRN-U 共设置九个主闸门。

| 闸门 | 位置 | 核心问题 | 通过后 |
|---|---|---|---|
| G0 | 项目入口 | 输入是否属于正确项目且来源明确 | 进入A层 |
| G1 | A→B | ProjectSeed 是否可启动 | 建立初始状态 |
| G2 | B→C | 当前状态是否有效、无阻断冲突 | 进入路径规划 |
| G3 | C→D | 是否存在且只存在一条合法主路径 | 进入章节设计 |
| G4 | D→E/F | 章纲是否可执行且不越权 | 进入正文生成 |
| G5 | E/F→G | 草稿是否形成可审计完整版本 | 进入独立审计 |
| G6 | G→H-P1 | 审计是否PASS且精确指向草稿版本 | 批准草稿 |
| G7 | B-UPDATE→H-P2 | 新状态是否可追溯且无冲突 | 创建快照 |
| G8 | H-P2→下一轮 | 快照链是否完整且冻结成功 | 进入下一章 |

# 6. G0｜项目与来源闸门

## 输入

```text
正式核心对象
或
尚未对象化的原始输入 + 当前项目上下文 / 新项目声明
```

## 必须检查

对于正式对象：

- project_id 是否存在；
- 对象类型是否明确；
- 版本是否明确；
- 来源是否可访问；
- 是否属于当前项目；
- 是否存在同名真源冲突；
- 是否是历史版本或当前版本。

对于新项目原始输入：

- 是否明确声明创建新项目；
- 是否成功分配唯一 project_id；
- 是否能够进入A层形成 ProjectBrief。

对于既有项目原始材料：

- 是否已经绑定明确的当前 project_id；
- 是否能够记录材料来源与版本。

## PASS

```text
新项目输入
→ 分配 project_id
→ A层创建 ProjectBrief

既有对象或材料
→ 进入对应责任层
```

## FAIL

```text
BLOCKED
→ H层真源核查
```

# 7. G1｜项目种子启动闸门

## 输入

```text
ProjectBrief
ProjectSeed
```

## 必须检查

- ProjectSeed.status 是否为 approved 或 locked；
- 一句话卖点是否明确；
- 内容源是否能够持续产生冲突；
- 主导人物结构是否明确；
- 长期冲突是否成立；
- 最小世界规则是否足以启动；
- 必须项与禁止项是否冲突；
- 第一阶段目标是否可执行。

## PASS

```text
→ B-INIT
```

## FAIL

```text
输入冲突 → A层
缺少人工选择 → 人工审批
续写材料缺失 → BLOCKED
```

# 8. G2｜叙事状态有效性闸门

## 输入

```text
NarrativeState
CharacterState 集合
ProjectSeed
上一 StateSnapshot（连续创作时）
```

## 必须检查

- B-INIT 已完成初始状态校验；
- 初始 NarrativeState 在交给C层前已由B层从 `initial` 转为 `active`；
- 同一项目只能有一个当前 active 状态；
- 关键事实有来源；
- 世界规则无阻断冲突；
- 核心角色当前目标可解释；
- 人物知识边界明确；
- 能力、资源、伤势和代价连续；
- 不可逆变化得到继承；
- 开放问题和叙事债务没有身份混乱；
- 不存在跨项目引用。

## PASS

```text
→ C层
```

## FAIL

```text
状态内容错误 → B层
快照链错误 → H层
来源冲突 → BLOCKED / 人工裁决
```

---

# 9. G3｜候选路径合法性闸门

## 输入

```text
CandidatePath 集合
NarrativeState
CharacterState 集合
ProjectSeed
```

## 每条路径必须检查

```text
事实一致
世界规则允许
人物愿意或被迫采取行动
人物拥有必要知识
能力与资源成立
前置条件成立
不可逆后果得到继承
服务当前阶段目标
```

## 路径状态规则

```text
硬约束不成立 → invalid
合法但未选择 → rejected 或 latent
本轮执行路径 → selected
```

同一 planning_round：

```text
selected 数量必须恰好为1
```

## PASS

```text
→ D层
```

## FAIL

```text
全部路径invalid → C层重新生成
状态导致无路可走 → B层
阶段目标本身错误 → A层 / 人工裁决
多条selected → C层修正
```

---

# 10. G4｜章节蓝图可执行性闸门

## 输入

```text
ChapterBlueprint
CandidatePath.selected
NarrativeState
CharacterState 集合
```

## 必须检查

- 蓝图引用正确的 selected Path；
- 本章主目标唯一；
- 起始状态与当前状态一致；
- 结束状态可由场景行动到达；
- 主要行动者明确；
- 核心阻力成立；
- 关键选择符合人物；
- 信息计划不泄漏知识；
- 场景顺序具备因果依赖；
- 章末压力来自本章后果；
- must_preserve 与 must_avoid 不冲突；
- acceptance_checks 可验证。

## 关于不可逆变化

当前冻结基线要求每章存在不可逆变化。

M0-04不自行修改该要求。

若过渡章实测无法合理满足：

```text
记录为上游规则风险
→ M1测试
→ 发起M0-01、M0-02、M0-03联合变更审查
```

## PASS

```text
ChapterBlueprint.status = approved
→ E层与F层
```

## FAIL

```text
场景结构问题 → D层
路径不成立 → C层
状态冲突 → B层
```

---

# 11. G5｜草稿可审计性闸门

## 输入

```text
ChapterDraft
ChapterBlueprint
```

## 必须检查

- ChapterDraft.status = review_ready；
- blueprint_ref 明确；
- object_version 明确；
- 正文内容完整；
- 不存在生成中断残片；
- known_deviations 已记录；
- changed_ranges 在重写时已记录；
- 字符编码和文件可读取；
- 审计对象能够精确定位段落或位置。

## PASS

```text
→ G层正式审计
```

## FAIL

```text
正文不完整 → E层
版本不明确 → H层版本核查
蓝图引用错误 → D层 / E层
```

---

# 12. G6｜审计通过闸门

## 输入

```text
AuditReport
ChapterDraft.review_ready
```

## 必须检查

- AuditReport.target_ref 精确匹配草稿版本；
- result = PASS；
- 不存在未解决硬约束问题；
- findings 与 result 不矛盾；
- 关键 unknowns 已处理或获准保留；
- protected_content 未被破坏；
- 复验要求已完成；
- 审计有明确证据位置。

## PASS

```text
→ H-P1校验版本匹配
→ ChapterDraft.status：review_ready → approved
→ 调用B-UPDATE
```

## FAIL路由

### result = REWRITE

```text
→ E层生成新草稿版本
→ G5
→ G层复验
```

### result = REPLAN

```text
→ D层
必要时返回C层
```

### result = BLOCKED

```text
→ B层状态核查
或人工裁决
```

### 审计报告自身无效

```text
→ G层重做AuditReport
```

---

# 13. G7｜状态提交闸门

## 输入

```text
ChapterDraft.status = approved
AuditReport.result = PASS
B-UPDATE 输出的新 NarrativeState
B-UPDATE 输出的新 CharacterState 集合
状态变更清单
当前 active NarrativeState
```

## 必须检查

- AuditReport.target_ref 精确匹配批准草稿版本；
- 新事实全部来自批准正文或人工裁决；
- 不得写入未执行蓝图内容；
- 人物知识变化有明确事件依据；
- 资源与能力变化可追溯；
- 关系变化有正文事件支撑；
- 叙事债务新增、推进或兑现可解释；
- 不可逆变化得到继承；
- 新状态没有跨项目引用；
- 新状态没有与上一快照无依据冲突；
- 新 NarrativeState.status = initial；
- 当前旧 NarrativeState 在预提交阶段仍保持 active；
- 尚未创建本轮 StateSnapshot。

## PASS

```text
→ 形成预提交状态包
→ 进入G8
```

预提交状态包是H层运行时的临时事务内容，不是第十类核心对象。

## FAIL

```text
提取遗漏或错误 → B-UPDATE
正文与状态不一致 → G层复核
批准正文存在隐藏硬错 → BLOCKED
H-P1审批状态需要撤销 → 保留旧active状态，草稿进入人工复核
```

G7失败时不得创建快照，也不得替换旧活动状态。

# 14. G8｜快照预提交与原子提交闸门

## 输入

```text
G7通过的预提交状态包
ChapterDraft.status = approved
AuditReport.result = PASS
新 NarrativeState.status = initial
新 CharacterState 集合
当前 NarrativeState.status = active
上一 StateSnapshot（首章除外）
CandidatePath.status = selected
H层提交记录草案
```

## 提交前必须检查

- approved_draft_ref 将指向正确的批准草稿版本；
- audit_report_ref 将指向匹配该草稿的PASS报告；
- narrative_state_ref 将指向新的 initial NarrativeState；
- character_state_refs 完整；
- previous_snapshot_ref 连续，首章除外；
- rollback_scope 明确；
- 提交记录包含本轮路径、审计和状态变更来源；
- 当前仍只有一个 active NarrativeState；
- 尚不存在本轮冻结快照；
- 快照内容一旦生成即可保持不可变。

## PASS：H-P2原子提交

以下动作必须作为一个不可分割提交完成：

```text
1. 创建 StateSnapshot.status = frozen
2. 当前旧 NarrativeState：active → superseded
3. 新 NarrativeState：initial → active
4. CandidatePath：selected → executed
5. 更新章节工作流与提交记录
6. 发出下一轮启动信号
```

任一步失败时，整个提交不得部分生效。

## 提交后条件

- 同一项目恰好一个 active NarrativeState；
- 本轮恰好一个 frozen StateSnapshot；
- 快照引用全部有效；
- 旧状态和旧快照保持可追溯；
- C层只读取新 active NarrativeState。

## FAIL

```text
预提交内容错误 → H-P2 / B-UPDATE
状态内容错误 → B-UPDATE
引用链错误 → H层恢复
原子提交失败 → 回滚本次事务，保留旧active状态
无法恢复 → 人工裁决
```

失败时禁止留下半冻结快照或双active状态。

# 15. 软检查体系

软检查不会自动阻断系统，但必须留下可审计记录。

## S-01｜语言冗余

检查：

- 同义重复；
- 动作后重复解释情绪；
- 多段反复确认同一事实；
- 无功能句。

可能路由：

```text
警告
或 REWRITE
```

---

## S-02｜AI味风险

检查：

- 通用句式；
- 情绪命名过多；
- 人物声音趋同；
- 句法过度均匀；
- 解释完整得不自然；
- 场景细节可无损搬运。

不能仅凭“像AI”直接判死刑，必须给出具体文本证据。

---

## S-03｜节奏同质

检查：

- 全章始终同一速度；
- 没有推进与停顿差异；
- 爆点持续堆积；
- 余波不足；
- 解释段过长。

---

## S-04｜认知负荷

检查：

- 同章新角色过多；
- 新术语过多；
- 活动谜团过多；
- 场景切换过密；
- 读者必须同时记忆的关系过多。

---

## S-05｜章末压力不足

检查：

- 章末没有新后果；
- 章末只是截断句子；
- 新危机与本章因果无关；
- 已解决全部压力却没有新方向。

---

## S-06｜人物声音偏移

检查：

- 词汇习惯变化无依据；
- 观察重点突然改变；
- 所有人说话长度和语气相同；
- 角色在压力下没有延续既有行为模式。

---

## S-07｜潜伏路径积压

检查：

- latent 路径无触发条件；
- 长期未推进；
- 与当前主线失去联系；
- 回收成本持续上升。

---

## S-08｜叙事债务逾期

检查：

- 已建立承诺长期不推进；
- 重复暗示但不兑现；
- 兑现后没有状态变化；
- 读者已无理由继续等待。

# 17. 失败严重度

## L1｜警告

不影响当前事实和主路径，可继续运行。

示例：

```text
局部用词普通
章节名未定
轻度句式重复
```

处理：

```text
记录
→ 可选修订
```

## L2｜局部修复

问题局限于正文或局部场景，不需要重做路径。

处理：

```text
REWRITE
```

## L3｜结构重做

问题来自蓝图或路径，局部润色无效。

处理：

```text
REPLAN
```

## L4｜系统阻断

事实、状态、真源、规则或人工权限冲突。

处理：

```text
BLOCKED
```

严重度不能由低层优点抵消。

# 18. 失败路由原则

## 18.1 最近合法责任层

问题必须返回最接近且拥有修改权限的层。

```text
词句问题 → E
场景问题 → D
路径问题 → C
状态问题 → B
项目方向问题 → A
快照和版本问题 → H
```

## 18.2 不跨层伪修复

禁止：

```text
F-STATE → E层改句子
F-PATH → G层润色
F-BLUE → 增加形容词
F-SRC → 模型自行猜测
```

## 18.3 保护已成立内容

修复时必须标明：

```text
允许修改范围
保护内容
影响对象
复验位置
```

不得因为局部错误重写全部有效内容。

## 18.4 原路复验

修复完成后，必须回到原失败闸门重新检查。

示例：

```text
G4失败
→ D层修蓝图
→ 重新进入G4
```

不能跳过原闸门直接向后推进。

# 22. 回滚规则

## 22.1 可回滚对象

```text
ChapterDraft
NarrativeState
CharacterState
StateSnapshot之后产生的下游规划
```

## 22.2 不允许的“回滚”

回滚不能：

- 删除历史记录；
- 覆盖旧快照；
- 假装错误从未发生；
- 修改已经发布内容而不留痕；
- 只回滚正文、不回滚依赖状态。

## 22.3 回滚最小范围

回滚应从最早受污染对象开始。

示例：

```text
错误起源于ChapterBlueprint
→ 回滚该蓝图之后的Draft、Audit、State与Snapshot

错误起源于NarrativeState
→ 回滚该状态之后的Path、Blueprint、Draft、Audit与Snapshot
```

## 22.4 提交事务失败

若H-P2原子提交未完成：

```text
不得将其视为一次正式章节回滚
```

应撤销本次未完成事务：

- 删除或废弃尚未正式提交的临时内容；
- 保留旧 active NarrativeState；
- 不创建 frozen StateSnapshot；
- 不把 selected CandidatePath 标记为 executed；
- 记录提交失败原因。

## 22.5 正式回滚后

必须：

```text
保留旧对象
标记 invalidated 或 superseded
建立新版本链
重新经过原闸门
```

## 27.2 验收中发现并修正的问题

1. **新项目入口无法满足project_id要求**  
   原G0要求所有输入预先携带project_id，会导致全新创作无法进入系统。现允许“创建新项目”声明，由G0先分配project_id，再由A层创建ProjectBrief。

3. **因果约束作用范围过宽**  
   将“任何事件”收窄为进入路径、蓝图或正式状态的关键事件，避免把每个微小动作都结构化审计。

5. **G8在快照冻结后才检查快照有效性**  
   该顺序违反快照不可变。现改为先验证预提交内容，全部通过后才创建 `StateSnapshot.frozen`。

6. **H-P2可能产生半提交状态**  
   新增原子提交规则：快照创建、旧状态失效、新状态激活、路径转executed必须整体成功或整体不生效。

7. **提交失败与正式回滚混淆**  
   现区分“未完成事务撤销”和“已冻结历史的正式回滚”。

8. **G2未说明初始状态如何进入active**  
   现明确B-INIT校验完成后，在交给C层前将初始NarrativeState从initial转为active。

9. **F-COMMIT责任范围不完整**  
   已扩展为B-UPDATE与H共同负责状态编译、预提交和原子提交错误。

10. **测试集缺少新项目入口、双active与事务半失败场景**

2. “每章必须存在不可逆变化”仍是待M1实测的上游风险，本文件不擅自修改。
3. M0-02的NarrativeState状态集合没有 `pending_commit`；本文件复用 `initial` 表示B-UPDATE产生但尚未提交的新状态，后续Schema设计时需验证该语义是否足够清晰。

禁止直接覆盖本文件。

# 28. 当前结论

SRN-U 的约束执行顺序为：

```text
先检查项目与真源
→ 再检查状态
→ 再检查路径
→ 再检查章纲
→ 再生成正文
→ 再独立审计
→ 再编译预提交状态
→ 最后原子切换状态并冻结快照
```

系统不以“写得顺”作为通过依据，而以：

```text
事实可信
人物成立
因果成立
路径合法
章纲可执行
正文可审计
审计已通过
状态可追溯
快照可回滚
```

作为最低运行条件。

当前文件只定义约束、闸门和失败路由，不代表相关检测器已经实现。
