# 4. M0核心基线

M0冻结后的SRN-U定义为：

> SRN-U是一个独立、通用、状态驱动的工业化小说创作引擎。

其最小可信闭环为：

```text
创作需求
→ 已批准项目种子
→ 初始叙事状态
→ 合法候选路径
→ 已选择主路径
→ 可执行章节蓝图
→ 正文草稿
→ 独立审计
→ 新状态预提交
→ 原子提交
→ 冻结快照
→ 下一轮规划
```

任何后续实现不得绕过：

```text
路径选择
正文审计
状态回写
快照冻结
```

# 5. 九类核心数据对象基线

M0冻结的九类对象为：

```text
1. ProjectBrief
2. ProjectSeed
3. NarrativeState
4. CharacterState
5. CandidatePath
6. ChapterBlueprint
7. ChapterDraft
8. AuditReport
9. StateSnapshot
```

对象身份基线：

| 对象 | 唯一职责 |
|---|---|
| ProjectBrief | 保存原始创作需求 |
| ProjectSeed | 保存已批准小说种子 |
| NarrativeState | 保存当前规范叙事状态 |
| CharacterState | 保存角色当前动态状态 |
| CandidatePath | 保存尚未执行的候选剧情路径 |
| ChapterBlueprint | 保存已批准的章节计划 |
| ChapterDraft | 保存具体正文版本 |
| AuditReport | 保存独立审计与修复路由 |
| StateSnapshot | 保存通过验收后的不可变检查点 |

冻结原则：

```text
事实 ≠ 计划
计划 ≠ 候选
候选 ≠ 草稿
草稿 ≠ 冻结事实
```

# 6. 八层模块基线

M0冻结的八层为：

```text
A 创作入口层
B 叙事状态层
C 结构规划层
D 章节设计层
E 正文执行层
F 生成控制层
G 审计修订层
H 连载运行层
```

职责基线：

| 层 | 核心职责 |
|---|---|
| A | 把原始需求编译为已批准项目种子 |
| B | 建立和更新规范叙事状态 |
| C | 生成候选路径并选择主路径 |
| D | 把主路径转换为章节蓝图 |
| E | 把蓝图写成具体正文 |
| F | 监测生成中偏差并提出控制信号 |
| G | 独立审计、定位错误、路由修复和复验 |
| H | 审批草稿、协调状态提交、冻结快照、回滚与继续 |

B层模式：

```text
B-INIT：建立初始状态
B-UPDATE：编译下一状态
```

H层模式：

```text
H-P1：校验PASS并批准指定草稿
H-P2：原子提交并冻结快照
```

# 7. 约束与闸门基线

## 7.1 全局优先级

```text
P1 已发生事实与冻结真源
P2 世界硬规则
P3 人物核心动机、人格与知识边界
P4 因果、能力来源与资源条件
P5 不可逆后果和已承担代价
P6 当前主路径与章节目标
P7 信息隐藏、释放与悬念
P8 语言、节奏和文风
P9 符号、传播与平台适配
```

低层优化不得覆盖高层约束。

## 7.2 九个主闸门

```text
G0 项目与来源
G1 项目种子启动
G2 叙事状态有效性
G3 候选路径合法性
G4 章节蓝图可执行性
G5 草稿可审计性
G6 审计通过
G7 新状态预提交
G8 快照预提交与原子提交
```

## 7.3 四类审计结果

```text
PASS
REWRITE
REPLAN
BLOCKED
```

## 7.4 原子提交基线

```text
旧状态保持active
新状态保持initial
↓
G7与G8全部通过
↓
H-P2原子提交：
旧状态→superseded
新状态→active
创建frozen快照
selected路径→executed
```

任一步失败，整个提交不得部分生效。

# 12. M0已知遗留问题

## RISK-01｜每章必须有不可逆变化

当前冻结基线要求每章有不可逆变化。

风险：

```text
可能对过渡章、余波章、关系章或纯观察章过强
```

处理：

```text
M1真实章节测试
→ 若出现系统性不适配
→ 发起M0-01至M0-05联合变更审查，并同步更新M0-06基线清单
```

当前状态：

```text
保留，不修改
```

---

## RISK-02｜AuditReport对象status枚举缺失

M0-02示例使用：

```text
status: final
```

但没有冻结AuditReport对象状态枚举。

当前工程统一只依赖：

```text
AuditReport.result
```

处理：

```text
M1-01 Schema设计时先形成正式变更决议
→ 更新受影响冻结文件
→ 再生成可执行Schema
```

在决议完成前，不得自行猜测或把 `final` 写成正式枚举。

---

## RISK-03｜CharacterState业务状态与版本当前性混用

CharacterState使用：

```text
active
inactive
missing
dead
superseded
archived
```

其中：

- `dead`属于角色业务状态；
- `superseded`属于对象版本状态；
- `active`可能同时表达角色活跃度和当前版本。

风险：

```text
Schema实现时可能产生语义冲突
```

处理：

```text
M1-01 Schema阶段必须先决议：
character_lifecycle_status
object_workflow_status
或给出等价且无歧义的方案
```

该问题解决前，不得把现有 `status` 直接编码为单一枚举。

---

## RISK-04｜NarrativeState使用initial表示待提交状态

M0-04复用：

```text
NarrativeState.status = initial
```

表示B-UPDATE已生成但尚未提交的新状态。

风险：

```text
“项目初始状态”
与
“预提交状态”
语义可能混淆
```

处理：

```text
M1-01 Schema阶段必须决定：
继续复用initial
或
新增pending_commit
```

任何语义选择或新增枚举都必须形成正式变更记录，并更新受影响测试。

---

## RISK-05｜测试样本不是完整Schema实例

```text
字段类型完整

```

处理：

```text
M1先建立Schema

```
