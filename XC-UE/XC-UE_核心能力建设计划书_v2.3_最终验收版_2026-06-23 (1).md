## 三、总体目标

最终形成以下可运行链路：

```text
加载小说项目
→ 读取项目状态和正文
→ L1 检测并产生证据
→ L1.5/L2 路由
```

- 安装 wheel 后可在仓库外运行核心流程；
- 业务规则由结构化文件控制；
- Markdown 只承担说明作用；
- 关键行为均有自动化测试证明。

# 阶段 A：平台基础闭合

## A1. 最小 Project Loader

### 目标

解除代码对 TP-001 路径的硬编码，但不删除、不迁移、不停用 TP-001。

TP-001 必须继续作为：

- 默认项目；
- 回归测试项目；
- 原有命令兼容对象。

### 实现范围

1. 新增结构化项目注册表；
2. 新增 Project Loader；
3. Project Loader 返回项目描述对象，而不是单一路径字符串；
4. 项目描述至少包含：
   - `project_id`；
   - `project_root`；
   - `harness_root`；
   - 必要入口和资源路径；
5. 替换统一入口、流水线、L1 默认项目逻辑及 L3 harness 参数中的 TP-001 路径硬编码；
6. 未指定项目时通过注册表加载默认 TP-001；
7. 显式指定 TP-001 时行为保持一致。

### 强制验收

- 现有 TP-001 命令兼容；
- 未指定项目时仍加载 TP-001；
- 显式指定 TP-001 时结果一致；
- 新增第二个测试项目时，只修改项目注册表，不修改流水线 Python；
- 未知项目返回结构化 `INPUT_INVALID`；
- 项目目录缺失时明确失败；
- 项目路径越界时明确拒绝；
- 所有失败必须发生在创建运行记录之前；
- 流水线实际传给 L3 的 harness 来自 Project Loader；
- 流水线、L1、L3 不得各自保留 TP-001 业务 fallback；
- 原有 TP-001 回归测试全部通过。

### 防伪验收

仅将硬编码从多个文件搬到一个加载器，不视为完成。必须证明新增项目不需要修改执行层 Python。

### 完成标志

> TP-001 从“源码写死的唯一项目”变为“默认可加载项目”，且原有使用方式不失效。

## A3. 业务规则结构化

### 控制权原则

```text
结构化 JSON/YAML：唯一运行真源
Python：加载、校验和执行
Markdown：说明，不参与运行决策
```

### 迁移范围

- 失败类型；
- 允许动作；
- 禁止项；
- 验收条件；
- 规则版本；
- 规则哈希；
- 规则状态和适用范围。

### 强制验收

- 规则文件有 Schema；
- 输出记录 `rule_id`、`rule_version`、`rule_hash`；
- 修改结构化规则会改变运行结果；
- 修改 Markdown 不会改变运行结果；
- 规则缺失、解析失败、Schema 错误时硬失败；
- 不允许静默使用 Python 默认业务规则；
- 不存在业务级 Python fallback；
- 原有路由和禁止项回归测试继续通过。

### 完成标志

> 所有运行时业务决策均可追溯到结构化规则，不再依赖 Markdown 标题或列表解析。

# 阶段 B：第一个真实业务闭环

本阶段只建设一个 L2 能力试点，不同时扩展六个能力。

首选：`L2-01 叙事结构能力`。

## B1. L2-01 真实诊断能力

### 输入

- L1 失败项；
- 失败证据；
- 正文定位；
- 项目上下文；
- 结构化能力规则。

### 输出最低要求

- 问题类型；
- 证据锚点；
- 涉及段落；
- 原因诊断；
- 修改目标；
- 候选修改策略；
- 风险；
- 验收条件；
- 置信度；
- 自动修复资格判定。

禁止仅生成通用模板式修复单。

### 强制验收样例

至少包含：

1. 正例：存在明确叙事结构问题；
2. 反例：不存在该问题，不得生成修复；
3. 证据不足样例：返回证据不足或人工判断；
4. 越界样例：命中禁止项后不得生成候选修改；
5. 差异样例：不同正文问题必须产生不同诊断和策略。

### 强制验收条件

- 所有文本锚点可在输入正文中复现；
- 无证据时不得伪造定位；
- 候选策略必须与问题类型对应；
- 不得仅替换模板字段；
- 能区分：
  - 可自动修复；
  - 需人工判断；
  - 禁止处理；
- 输出满足 Schema；
- 专项测试和全量 pytest 通过。

---

## B2. L3 最小执行闭环

### 目标流程

```text
读取 L2 候选修复
→ 创建隔离工作副本
→ 执行确定性修改
→ 生成统一 diff
→ 重跑相关 L1 闸门
→ 输出复验结果
→ 等待审批
→ 原子应用或回滚
```

### 状态机

正常路径：

```text
PLANNED
→ EXECUTED_IN_SANDBOX
→ REVALIDATED
→ AWAITING_APPROVAL
→ APPLIED / REJECTED / ROLLED_BACK
```

失败终态：

```text
EXECUTION_FAILED
REVALIDATION_FAILED
APPLY_FAILED
ROLLBACK_FAILED
ABORTED
```

任何异常都必须进入明确终态，不得永久停留在中间状态。

### 默认安全边界

- 审批前不得修改原文；
- 不自动批准；
- 执行失败必须保留原始文件；
- 回滚不依赖人工恢复；
- 同一任务重复执行不得重复修改。

### 强制验收

- 至少一个 TP-001 样例完成端到端闭环；
- 修改前后正文均有哈希；
- 审批前原文件字节和哈希完全不变；
- diff 与实际候选文件一致；
- 复验针对候选版本，而不是旧原文；
- 复验结果绑定候选版本哈希；
- 拒绝审批后原文不变；
- 应用过程任一步失败时可恢复；
- `APPLY_FAILED` 后必须自动尝试回滚；
- 回滚成功后文件哈希等于修改前哈希；
- 回滚失败必须进入 `ROLLBACK_FAILED`，不得伪装成已恢复；
- 执行失败进入 `EXECUTION_FAILED`；
- 复验失败进入 `REVALIDATION_FAILED`；
- 主动终止进入 `ABORTED`；
- 任意异常不得永久停留在 `PLANNED`、`EXECUTED_IN_SANDBOX`、`REVALIDATED` 或 `AWAITING_APPROVAL`；
- 每个失败终态必须记录错误原因、最后完整产物哈希和最后成功阶段；
- 同一任务重复执行具备幂等保护；
- 专项测试和全量 pytest 通过。

### 阶段 B 完成标志

> XC-UE 首次具备从问题证据到候选修改、diff、复验、审批、应用和回滚的真实垂直闭环。

# 阶段 D：生产就绪

本阶段只回答一个问题：

> 当前已经形成的引擎能力，是否具备受控进入生产使用的条件。

本阶段不承担 L2-02 至 L2-06 的能力扩建，但不能仅依赖 TP-001 测试夹具宣布生产就绪。至少必须在一个非 TP-001 的真实或脱敏项目上完成受控运行验证。

## D1. 生产规则集

### 目标

- 建立明确版本的生产规则；
- 区分候选规则和生产规则；
- 记录规则版本、哈希、状态和适用范围；
- 规则升级可回滚；
- 不以 Markdown 状态文字代替运行时校验。

### 强制验收

- 生产规则具有明确版本和可机读状态；
- 生产模式只能加载允许状态的规则；
- 规则哈希进入运行血缘；
- 规则升级和回滚均有自动化测试；
- 无生产规则时明确拒绝生产运行；
- 候选规则不得静默进入生产模式；
- 生产规则缺失、损坏或 Schema 不合格时硬失败。

## D2. 版本兼容与升级回滚

### 目标

建立引擎、项目格式、Schema 和规则版本之间的兼容边界。

### 强制验收

- 引擎版本、项目格式版本、Schema 版本和规则版本均可追溯；
- 不兼容版本组合明确拒绝运行；
- 升级前后有可重复的迁移或兼容测试；
- 引擎升级失败时可以回退；
- 规则升级失败时可以恢复原规则版本；
- 生产运行不得依赖未声明的隐式兼容行为。

## D3. 生产运行安全

### 强制验收

- 生产运行失败后原始正文保持可恢复；
- 运行记录、候选产物、审批结果和最终应用结果有完整血缘；
- 关键写入采用原子操作；
- 中断、超时和异常退出均有明确最终状态；
- 重试不会重复应用同一修改；
- 仓库外安装环境可完成正式流水线；
- 至少一个非 TP-001 项目完成正式流水线；
- 至少一个真实或脱敏稿件样例完成候选修改、复验、审批和应用/拒绝流程；
- 不允许仅以 TP-001 测试夹具作为生产就绪证据；
- 该项目接入不得修改执行层 Python；
- 发布门槛由自动化检查证明，不由人工口头声明替代。

### 阶段 D 完成标志

> 已有 L2-01 垂直能力可以在至少一个非 TP-001 的真实或脱敏项目上，按明确版本、明确规则、明确回滚和明确失败恢复边界受控运行。

## 六、统一完成标准

每项工程任务统一采用：

```text
代码修改
→ 专项测试
→ 相关层回归测试
→ 全量 pytest
→ git diff --check
→ 独立 Git 提交
```

完成报告只包含：

- 修改文件；
- 关键行为变化；
- 专项测试结果；
- 全量测试结果；
