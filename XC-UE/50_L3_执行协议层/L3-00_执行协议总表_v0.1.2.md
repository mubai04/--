# L3-00 执行协议总表 v0.1.2

> 状态：候选真源 / 未定版 / 待测试  
> 层级：L3 执行协议层  
> 文件性质：执行入口总表  
> 真源规则：Markdown 为真源；图片、对话、临时指令不得覆盖本文件。  
> 版本说明：v0.1.2 当前默认 `L3` 入口冻结为 TASK_PLANNING_ONLY：只生成任务包、分项任务、任务依赖、约束、目标文件引用与复验入口；不生成候选正文、不生成 diff、不声明回填完成。
>
> 补充说明：上述冻结范围仅适用于默认 `L3` 任务规划入口，不覆盖 B2 的受限 `L3_PATCH` 运行时执行线。`L3_PATCH` 只允许在 `protocol_rules.json` 的 `patch_execution` 启用、人工审批明确通过、最终决定绑定、补丁内容冻结、`source_sha256`/`candidate_sha256`/`patch_sha256` 匹配、目标仍为 `TP-001`、策略仍为 `L2-01` 的单一确定性 `REPLACE`/`APPEND`、且候选正文已经通过 L1 复验后，对正式正文执行原子应用或自动回滚；未经批准不得修改正式正文的硬闸门保持不变。

## 0. L3 定位

L3 不是新的创作理论层。

L3 是任务规划协议层，负责把上游判断变成标准任务包：

L1 判断结果  
→ L1.5 路由结果  
→ L2 修复方向  
→ L3 任务规划  
→ 任务包落盘  
→ 等待人工或后续执行器  
→ 回 L1 / L1.5 复验入口

一句话：

> L3 = 判断结果到受约束任务包的规划层。

---

## 1. L3 负责什么

L3 负责：

1. 接收 L1 / L1.5 / L2 / L2-99 的执行输入。
2. 转译为人工或后续执行器可读取的任务规划。
3. 指定读取文件、目标文件引用、任务依赖、约束和复验入口。
4. 规定正文修复任务包的字段格式。
5. 规定禁止修改范围和任务阻断记录。
6. 规定修复后应回到哪里复验，但不声明回填完成。
7. 规定任务规划失败如何记录断点。
8. 规定 IR 输入如何映射到执行任务。
9. 规定 Project Harness 如何作为测试项目壳运行。

---

## 2. L3 不负责什么

L3 不负责：

1. 判断作品好不好。
2. 判断读者是否投入。
3. 判断发布锁是否通过。
4. 判断主失败归属。
5. 替 L1.5 路由。
6. 替 L2 做能力判断。
7. 直接覆盖 Markdown 真源。
8. 无任务单直接改正文。
9. 生成候选正文、diff、备份或回填完成记录。
10. 生成新系统层。
11. 把测试项目壳当 XC-UE 系统真源。
12. 把执行成功等同于 L1 / L2 判断通过。

## 3. L3 任务规划状态机

<!-- XCUE:L3_STATE_MACHINE:START -->
所有 L3 任务必须落入以下状态机：

```text
RECEIVED
↓
INPUT_VALIDATED
↓
TASK_PLANNED
↓
TASK_PACKAGE_CREATED
↓
AWAITING_EXECUTOR
↓
EXECUTION_STARTED
↓
EXECUTION_COMPLETED
↓
ACCEPTANCE_PENDING
↓
ACCEPTED
```

异常状态：

```text
VALIDATION_FAILED
BLOCKED
EXECUTION_FAILED
ROLLED_BACK
```

L3 任务不得跳过：

- 输入校验
- 权限和目标引用校验
- 任务依赖记录
- 约束记录
- 回流验收位置

当前 v0.1.2 只允许进入 `TASK_PACKAGE_CREATED` 与 `AWAITING_EXECUTOR`，不得把任务包生成标记为 `EXECUTION_COMPLETED`。
<!-- XCUE:L3_STATE_MACHINE:END -->

### 3.1 B2 L3_PATCH 状态机映射

`L3_PATCH` 使用单独的 B2 运行时状态机，由 `protocol_rules.json` 与 `第三层补丁审计结构.json` 共同约束。

| 计划书状态 | B2 实现状态 | 含义 |
|---|---|---|
| `PLANNED` | `PENDING_APPROVAL` | 已生成确定性补丁计划，等待候选阶段审批 |
| `REVALIDATED` | `READY_FOR_ACCEPTANCE` | 候选正文、diff 与 L1 复验血缘通过 |
| `AWAITING_APPROVAL` | `READY_FOR_ACCEPTANCE` + 最终决定缺失 | 等待人工最终 apply/reject |
| 最终采纳 | `APPLIED` | 正式正文已原子应用并通过应用后复验 |
| 最终拒绝 | `REJECTED` | 正式正文未改变 |
| 异常恢复 | `ROLLED_BACK` | 应用后异常已回滚至原始哈希 |
| 异常终态 | `APPLY_FAILED` / `ROLLBACK_FAILED` / `EXECUTION_FAILED` / `REVALIDATION_FAILED` / `ABORTED` | 失败终态，必须记录错误、最后成功阶段和最终正文哈希 |

允许迁移：

```text
PENDING_APPROVAL -> EXECUTED_IN_SANDBOX -> READY_FOR_ACCEPTANCE
READY_FOR_ACCEPTANCE -> APPLIED
READY_FOR_ACCEPTANCE -> REJECTED
APPLIED/REJECTED -> 只允许幂等返回既有互斥终态
应用后异常 -> ROLLED_BACK 或 ROLLBACK_FAILED
```

禁止迁移：

```text
APPLIED -> REJECTED
REJECTED -> APPLIED
READY_FOR_ACCEPTANCE -> APPLIED（缺最终决定、缺备份、缺原子写、缺应用后复验时）
任意状态 -> 非 TP-001 / 非 L2-01 / 非 REPLACE|APPEND
```

每次 L3 任务规划后必须输出任务摘要：

```yaml
任务摘要:
  规划了什么: ""
  为什么规划: ""
  依据哪个修复单: ""
  任务包文件: ""
  分项任务文件: ""
  任务依赖: []
  约束: []
```

## 8. L3 执行顺序

<!-- XCUE:L3_EXECUTION_ORDER:START -->
1. 校验任务来源。
2. 校验 IR / Project Harness 输入。
3. 校验目标文件。
4. 校验是否允许改正式正文。
5. 校验权限矩阵。
6. 生成任务包和分项任务。
7. 记录任务依赖。
8. 记录约束和禁止修改文件。
9. 记录目标文件引用。
10. 标记复验入口。
11. 若失败，写入断点记录。
<!-- XCUE:L3_EXECUTION_ORDER:END -->

## 9. L3 最小验收

L3 通过标准：

1. 能明确接收上游任务。
2. 能明确知道读哪些文件。
3. 能明确知道目标文件引用。
4. 能明确知道禁止改哪些文件。
5. 能明确知道任务包写到哪里。
6. 能明确记录任务依赖。
7. 能明确记录任务约束。
8. 能明确知道回哪里复验。
9. 能明确记录断点。
10. 不越权判断内容质量。
11. 任务包写入成功不等于 L1 / L2 复验通过。
