## 一、总判定

XC-UE 已经不再是纯 Markdown 空壳。当前版本包含实际 Python 执行代码、统一入口、L1/L2/L3 流水线、JSON Schema、运行报告、路径安全检查和自动化测试。

但它目前仍不能被判定为“生产级小说工程引擎”或“完整执行闭环”。更准确的定位是：

> **候选态启发式正文审查器 + 修复任务规划器 + 单一测试项目适配层。**

它已经能做三件事：

1. 对候选正文做启发式筛查；
2. 根据筛查结果生成修复单和工程任务；
3. 在无生产规则时拒绝进入生产模式。

它还不能可靠完成五件关键事项：

1. 通过统一入口稳定运行全部公开目标；
2. 对任意外部项目进行注册、装载和执行；
3. 保证 Markdown 规则是真正控制运行的唯一真源；
4. 执行修复、应用改动并自动回流复验；
5. 以经过校准的指标证明“读者投入”“发布可行性”等业务判断有效。

因此，当前版本的核心问题不是“完全没有代码”，而是：

> **代码、文档、状态语义、项目适配和实际能力之间尚未形成同一个可验证闭环。**

# 三、P0：阻断项目成立的问题

## P0-01：统一入口公开了一个必然失败的 TP-001 调用路径

### 现象

执行：

```bash
python 00_工程总控/工程执行层/统一运行入口.py --target TP-001 --run-id AUDIT-TP001-001
```

统一入口会强制向 TP-001 子入口附加：

```text
--standard-mode CANDIDATE_TEST
```

但 `TP001运行入口.py` 并不接受该参数，最终 argparse 返回退出码 2。

### 判定

这是确定性接口契约错误，不是环境问题。

### 影响

- 文档或统一入口宣称支持的目标实际不可用；
- 当前测试没有覆盖公开 CLI 契约；
- 后续任何上层调度器调用 TP-001 都会失败。

### 必须整改

1. 统一定义所有 target 的参数契约；
2. 统一入口只能传递子入口明确声明支持的参数；
3. 为每一个公开 target 增加 CLI 冒烟测试；
4. 文档中的每条命令必须进入自动化测试。

### 验收标准

- `--target TP-001` 返回 0；
- 未支持参数返回项目自定义错误，而不是裸 argparse 错误；
- CLI 文档示例全部由测试逐条执行。

## P0-02：项目没有形成“外部项目注册—装载—运行”机制

### 现象

主流水线直接硬编码：

```text
70_测试项目/TP-001_CleanHarness_IR_Runtime
```

L1 默认正文路径也绑定到 TP-001 候选正文。命令行中的 `--project` 主要只是标签，不是实际装载项目的定位参数。

当前没有可用的：

- 项目注册表；
- 项目清单解析；
- `--project-manifest` 或 `--project-root`；
- 项目能力/规则兼容性校验；
- 多项目隔离测试。

### 判定

XC-UE 仍然是“引擎代码 + TP-001 特制接线”，不是中立的独立引擎。

### 影响

接入第二个项目时必须修改引擎代码。项目与引擎无法独立升级，也无法证明不存在跨项目污染。

### 必须整改

建立最小项目注册协议，例如：

```yaml
project_id: TP-001
project_root: ...
chapter_source: ...
formal_source: ...
ruleset: ...
output_root: ...
allowed_capabilities: ...
```

统一入口必须根据 manifest 装载项目，而不是引用 TP-001 固定目录。

### 验收标准

- 不修改 XC-UE 源代码即可注册第二个测试项目；
- 两个项目可分别运行，产物目录互不污染；
- 删除 TP-001 后，引擎本体仍能启动并给出“未注册项目”的规范错误；
- 代码扫描中不存在业务项目固定路径。

## P0-03：Markdown“真源”声明与实际运行控制源不一致

### 现象

根文档宣称 Markdown/Front Matter 是工程真源，但关键判断实际硬编码在 Python 中，包括：

- L1 触发词和模式；
- E/V/C 评分规则；
- `E=3`、`V=3`、`C_max=3`、`I_min=6` 等阈值；
- L2 默认路由；
- 默认修复动作；
- 标准文件名和版本；
- 若干状态映射。

运行时没有看到一条完整链路证明：

```text
Markdown 规则 → 结构化解析 → Schema 校验 → 运行时装载 → 判定执行
```

### 判定

当前实际存在“双真源”：文档是真源声明，Python 是真实行为源。

### 影响

- 修改 Markdown 不一定改变系统行为；
- 修改 Python 可能悄然改变规则而文档不变；
- 审查者无法从 Markdown 重现运行结果；
- 所谓“冻结规则”不能约束执行代码。

### 必须整改

二选一，不能继续模糊：

**方案 A：结构化规则是真源。** 将阈值、模式、路由和状态映射迁入 YAML/JSON，Markdown 只解释；Python 只负责执行。

**方案 B：代码是真源。** 明确取消 Markdown 运行真源声明，文档通过代码生成或校验。

更适合当前项目的是方案 A，但必须缩小范围，先只迁移 L1 和路由表，不要一次改造全部文档。

### 验收标准

- 改变规则文件中的阈值后，测试结果同步改变；
- Python 中不再出现重复阈值和业务正则；
- 每次运行记录规则文件哈希和解析版本；
- 未通过 Schema 的规则禁止运行。

## P0-04：所谓“完整闭环”在 L3 处中止

### 现象

L3 输出明确写着：

```text
execution_mode: TASK_PLANNING_ONLY
task_package_created: false
awaiting_executor: true
```

L3 当前只生成任务规划，不执行修复，不生成差异，不应用修改，也不自动回流 L1/L2 复验。

### 判定

这不是执行闭环，而是“检测—规划”半闭环。

### 影响

- 系统不能证明修复任务可执行；
- 无法验证任务执行后问题是否消失；
- 不能统计修复成功率、回归率和副作用；
- “工程执行层”命名超出实际能力。

### 必须整改

短期不要直接做正文自动改写。先建立可验证的执行器契约：

```text
任务包 → 执行器接收 → 生成候选差异 → 人工确认/沙箱应用 → 回原闸门复验 → 结果归档
```

执行器可以先只处理确定性的工程任务，例如路径、元数据、格式、缺失字段和报告生成。

### 验收标准

- 至少一种任务能被真实执行；
- 执行前后有 diff；
- 修复后自动回到原失败闸门；
- 回流失败会生成新状态，不会覆盖旧证据；
- L3 不再把“等待执行器”表述成已完成闭环。

## P0-05：顶层成功状态掩盖了“候选、未验证、无发布权限”

### 现象

候选流水线最终可返回：

```text
final_status: SCREENING_PASS
```

但同一运行内部同时包含：

```text
heuristic: true
publish_authority: false
human_review_required: true
validation_status: UNVALIDATED
standard_mode: CANDIDATE_TEST
```

L2 无失败项时生成空修复报告，L3 无任务时生成空任务束，二者仍标为 `COMPLETED`。

### 判定

状态字段各自未必错误，但顶层聚合语义具有误导性。

### 影响

外部调用者只读取 `SCREENING_PASS` 时，容易将“启发式未发现风险”误解为“质量通过”或“可发布”。

### 必须整改

建立不可混淆的状态层次：

```text
run_status             执行是否完成
screening_result       启发式筛查结果
validation_status      模型是否经过验证
publication_authority  是否具有发布授权
workflow_status        是否等待人工/执行器
```

候选模式顶层结果建议改为：

```text
CANDIDATE_SCREENING_NO_RISKS_FOUND
```

禁止使用无修饰的 `PASS` 表示候选启发式结果。

### 验收标准

- 任一候选运行的顶层状态都包含候选语义；
- `UNVALIDATED` 必须向顶层传播；
- `publish_authority=false` 时禁止出现发布通过状态；
- 空报告使用 `NOT_APPLICABLE` 或 `NO_ITEMS`，而不是笼统 `COMPLETED`。

## P0-06：正式规则集不存在，生产模式不能运行

### 现象

生产模式退出码 24：

```text
NO_PRODUCTION_RULESET: 生产模式没有 ACTIVE/FROZEN 规则集
```

24 份系统规则均处于 `CANDIDATE`，并且 `source_qualification` 为 false。

### 判定

生产拒绝本身是正确行为，但它证明项目当前不具备生产运行条件。

### 影响

任何“已经可以生产使用”的说法均不成立。

### 必须整改

不能直接把全部文档批量改成 ACTIVE/FROZEN。必须先完成：

1. 真源机制统一；
2. 规则依赖补全；
3. 测试映射；
4. 规则验证证据；
5. 最小生产规则集冻结。

### 验收标准

- 只冻结最小可运行规则集；
- 每条正式规则绑定测试编号和证据；
- 生产运行记录精确规则版本和哈希；
- 候选规则不得被生产入口加载。

## P0-07：根级控制文档引用了不存在的强制文件

### 现象

README、INDEX、AGENTS 和 L0 文档引用：

- `XC-UE_项目全量问题盘查报告_2026-06-23.md`

### 判定

根级索引目前不能作为可信导航和验收入口。

### 影响

Codex 或新维护者按根文档执行时会进入不存在路径，且无法确认哪个文件是正式控制文件。

### 必须整改

- 补齐真正需要的文件，或删除过时引用；
- 增加 Markdown 路径完整性测试；
- 根级入口只保留一个当前状态页和一个任务页；
- 禁止索引引用未来计划文件作为现存事实。

### 验收标准

- 根目录所有路径引用可解析；
- 所有文档图片可打开；
- CI 中加入链接/路径检查；
- INDEX 与实际文件树自动比对。

# 四、P1：严重但可在 P0 后处理的问题

## P1-01：Python 打包是“表面成功、实际空包”

`pip install . --no-deps` 可以执行成功，但生成的 wheel 约 1,014 字节，仅包含 `dist-info`，没有安装任何实际模块，也没有 console script。

`pyproject.toml` 因而制造了“可安装项目”的假象。

整改：将稳定代码迁入 `src/xcue/`，配置包发现、入口脚本和最小导入测试；在此之前删除或明确标注实验性打包配置。

---

## P1-02：同时存在三条重叠执行路径

当前至少有：

1. 主 L1/L2/L3 流水线；
2. 已标记弃用的“正文检测”兼容入口；
3. TP-001 本地引擎，包括拆分模块和疑似遗留单体 `tp001_engine.py`。

弃用入口仍被统一入口公开，并残留特定小说项目默认值。单体 TP-001 引擎与拆分版本职责重叠。

整改：确定一条唯一主链；兼容入口迁入 `legacy/`，默认关闭；重复实现删除或明确版本迁移关系。

---

## P1-03：L2 六项能力在代码层没有形成六种能力

`L2_01` 至 `L2_06` 六个 Python 文件基本是相同的 7 行包装器，全部调用同一个通用修复单生成逻辑。

而 L2 文档约 9,760 行，文档复杂度远高于实际执行差异。

当前 L2 的真实能力是“按类型生成模板化修复建议”，不是六个独立的文风、叙事、角色、设定、AI 味或市场修复器。

整改选择：

- 要么将模块改名为“六类修复路由模板”；
- 要么为每类能力建立独立输入、算法、输出和测试。

在未实现之前，不应继续增加 L2 文档体量。

---

## P1-04：L1 指标名称大于实际测量能力

L1 的 E/V/C/I 主要由关键词、正则命中、段落密度和固定阈值计算。它们可以作为启发式信号，但不能直接证明：

- 读者真的愿意继续阅读；
- 作品具备发布质量；
- 某种类型或平台上的留存有效。

目前缺少：

- 人工标注样本；
- 类型/平台分层；
- 精确率、召回率、误报率；
- 阈值校准；
- 与真实读者或编辑判断的相关性验证。

整改：保留指标，但将名称收缩为“启发式代理信号”，先建立小型人工标注验证集，再决定是否升级业务含义。

---

## P1-05：自动化测试数量可用，但覆盖面偏向集成链

37 个测试全部通过，这是正面证据。但测试主要依赖子进程和端到端调用，完整耗时 49.39 秒，单个慢测试约 13.15 秒。

缺失的关键测试包括：

- 统一入口所有 target；
- 文档 CLI 示例；
- 打包安装和导入；
- Markdown 链接完整性；
- 根级必需文件；
- 第二项目注册与隔离；
- 规则文件改动是否影响运行；
- Windows 实机回归；
- TP-001 Markdown 中声明的测试表自动化。

整改：拆分 `unit`、`integration`、`acceptance` 三层；日常快速测试应在数秒内完成，完整验收再运行子进程链。

---

## P1-06：规则依赖元数据近乎空置

24 份系统规则的以下字段普遍为空：

- 上游依赖；
- 下游引用；
- 替代关系。

这与实际存在的 L1→L1.5→L2→L3 关系不符。

结果是 Front Matter 虽然存在，但没有形成可计算规则图谱。

整改：只为机器真正需要解析的规则保留元数据，并通过脚本验证引用对象存在。不要人工维护无消费方的装饰字段。

---

## P1-07：README 对当前工程状态的描述已经失真

README 仍写有“当前所有内容仍处于 Markdown 文件态”“这还不是可执行工程本体”等表述，但仓库实际已有 80 个 Python 文件、流水线和自动化测试。

正确描述应是：

> 已形成实验性可执行骨架，但仅支持候选启发式筛查与任务规划，尚未达到生产闭环。

状态文档失真会导致后续人员重复建设或错误验收。

---

## P1-08：正式正文仍是占位文件

测试项目中的正式 `chapters/ch01.md`、`ch02.md` 仍是占位内容；可通过候选流水线的是候选正文，不是正式真源正文。

这意味着：

- 成功运行不能证明正式项目已完成接入；
- 候选与正式版本的接受、合并和回滚协议尚未落地；
- “正式正文保护”有基础，但“候选转正式”没有完整机制。

# 五、P2：工程质量与维护成本问题

`.gitignore` 虽然排除了这些内容，但发布归档仍然带入，说明打包流程不是从干净工作树生成。

## P2-03：存在疑似未使用模块和重复代码

如：

- `L3工程/日志记录校验.py`；

当前导入图中未见其成为主链必要节点。需要做静态引用核验后删除、接入或标记 legacy。

## P2-04：标识命名不完全一致

大多数 L3 文档使用 `XCUE-RULE-*` 命名空间，但 `L3-05` 使用 `XCUE-L3-005`。虽然没有发现重复 ID，但命名不一致会增加规则解析和迁移成本。

## P2-05：日志目录和运行记录治理不完整

`90_日志` 为空，但文档将其当作已有证据源引用。运行时产物和长期审计日志之间缺少：

- 保留策略；
- 归档策略；
- 敏感内容约束；
- 清理命令；
- 运行 ID 唯一性规则。

# 六、问题之间的根因关系

这些问题不是彼此独立的。根因链如下：

```text
文档先行且体量过大
        ↓
“真源”未转化为可执行规则装载
        ↓
Python 为了先跑起来而硬编码阈值、路径和路由
        ↓
TP-001 被写进引擎主链
        ↓
出现多入口、多实现和接口漂移
        ↓
测试验证了局部流水线，却没有验证公开契约和第二项目
        ↓
顶层仍能给出 SCREENING_PASS，形成能力错觉
```

因此，不能通过继续补 Markdown 或继续增加 L2/L3 模块来解决。当前应先收缩并统一工程主链。

# 十、反向压力测试

## 薄弱假设 1：TP-001 本来就只是独立样板，硬编码无所谓

反驳：如果统一入口、主流水线和默认正文都依赖该样板，那么它已经不是独立样板，而是运行时依赖。只有将其移除后主引擎仍可正常启动，才能证明它只是样板。

## 薄弱假设 2：Markdown 只是说明书，不需要控制运行

反驳：如果接受该前提，就必须删除“Markdown 真源”和冻结规则约束运行的声明。问题不在于一定要用 Markdown，而在于声明的治理模型与真实行为不一致。

## 薄弱假设 3：37 个测试通过足以证明项目可用

反驳：测试只能证明被覆盖的行为。公开 TP-001 CLI 仍然确定性失败、空 wheel 仍可构建、第二项目接入仍不存在，说明现有测试集合没有覆盖产品级契约。

最终结论不变：当前项目具备实验性工程骨架，但尚未形成中立、可安装、单一真源、可执行修复并能回流验证的生产闭环。
